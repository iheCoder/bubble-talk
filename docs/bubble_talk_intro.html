先看理念与架构
Manifesto

把对话变成第一公民

BubbleTalk 不是“内容页 + 对话框”，而是一台以会话为主轴的电影剪辑机：学习被重构为一段段可推进、可回放、可验收的对话现场。 我们强调两件事：一是原生语音多模态（例如基于 gpt-realtime 的流式 ASR/TTS 与插话中断），二是宏观导演的克制（在不依赖工具的纯对话里，仍能稳住叙事与目标）。

Principle 01

对话是主菜

学习推进以“你说一句，我回一句”的互动为主轴，而不是把一段课程塞进播放器。对话里的每个转折、每次追问、每次复述都能被系统看见并利用。

Principle 02

语音原生

不把语音当作“输入法”。它是节奏与情绪的载体：流式转写、实时打断（barge-in）、多角色音色切换，让研讨像现场发生。

Principle 03

导演的克制

宏观导演可以完全不用工具：只靠叙事结构与场面调度，把用户从跑题与无效闲聊里救回来；工具是加速器，不是拐杖。

“对话为第一公民”在工程上意味着什么？

  • 01会话事件流先行：所有关键输入/输出都追加到 Timeline，支持回放与验收。
  • 02单会话串行编排：避免“重复播报/重复出题/计划错位”。
  • 03决策与表达解耦：Director 只输出结构化计划，Actor 只负责编译 Prompt，Realtime 负责说话。
  • 04可控不漂移：硬约束（如 90 秒必须触发用户输出动作）优先于模型自由发挥。

The Problem 为什么现在的 AI 学习
依然让人觉得累？

⛰️

启动难 (Friction)

传统课程像“爬矿山”，需要整块时间与巨大意志力。未学先累，即使是 AI 生成的课件也难以改变“重”的本质。

😶

太被动 (Passive)

听播客、看视频，你只是“旁观者”。知识如过眼云烟，听完以为懂了，一用就忘。缺乏深度加工的机会。

🌫️

易跑偏 (Drift)

直接跟 ChatGPT 聊？很容易陷入无主题的闲聊，或者被 AI 的“幻觉”带偏，缺乏系统的教学结构。

双引擎导演系统

Dual-Engine Director System

01

宏观导演

Showrunner / Segment Director

负责“叙事结构”与“场面调度”。它像一位金牌导播，确保对话流畅、有趣且不跑题。

Script Alignment 机制

⛔️ 用户跑题
⚡️ Director Action: RESCUE & REWRITE
"修改后续剧本，顺着用户的话题自然引导回主线..."
✅ 故事继续
克制点：宏观导演优先用“转场、提问、重写叙事线”解决问题， 尽量不借助工具打断对话的流动性。
02

微观导演

Cognitive Coach / Beat Director

负责“教学效果”。实时监控你的认知负荷 (Load) 和掌握度 (Mastery)，动态调整对话难度。

Hard Constraint

90 秒必须让你“输出”一次

复述、判断、举例、迁移题、Exit Ticket… 导演不允许你一路点头到结束。

Branch Control

分叉问题入栈，主线不丢

允许插话，但把“好问题”放进待回收栈：先推进主线，再按策略回收。

Role Switching

同一段知识，多视角切入

主持人控节奏，专家给骨架，挑战者抛反例；你是特邀嘉宾，不是观众。

Architecture

不只是一个数据流

BubbleTalk 的关键不是“回答得像人”，而是“对话推进像一台可控的剪辑台”。下面是核心组件，以及它们如何把一次用户插话变成一次可回放、可验收的学习推进。

Realtime Gateway

实时语音网关

与 gpt-realtime 建连，双向音频流（ASR/TTS），负责插话中断与指令下发。

Session Orchestrator

会话编排器

事件 append-first、单会话串行、reduce 归约；把事实变成决策，把决策变成输出。

Director + Actor

导演 / 演员引擎

Director 产出结构化 DirectorPlan；Actor 编译成可校验的 Prompt 指令，Realtime 负责说。

Assessment

测评与练习引擎

自适应测评（2–4 题）、拍点内微测评、Exit Ticket；把“我懂了”变成可验证。

Knowledge & Assets

概念图谱与素材层

Concept Graph、Scenario Library、图解/漫画/卡片模板；多模态是配菜，但必须可追踪。

Telemetry

会话分析与回放

关键事件写入 Timeline；为学习模型更新、推荐与工程排障提供证据链。

一轮对话是如何“被剪辑”的

以“用户语音插话”为例：从事件流到 DirectorPlan，再到 Realtime 语音输出。

编排器方案 整体技术方案 一段真实 Flow Log
  1. 01 Event
    网关收到音频与 ASR final，转成标准化事件并交给编排器。
  2. 02 Timeline
    编排器 append-first 写入 Timeline，分配 seq，保证可回放与幂等。
  3. 03 Reduce
    归约 SessionState，更新 OutputClock、分叉栈、掌握度信号等。
  4. 04 Decide
    Director 根据面板输入输出 DirectorPlan（下一拍点/角色/输出动作/栈操作）。
  5. 05 Guardrails
    硬约束先行：例如 90 秒必须触发用户输出动作；用户说结束必须进入 Exit Ticket。
  6. 06 Speak
    Actor 编译 Prompt 指令，经网关送入 gpt-realtime，产生流式 TTS 与字幕，写回 Timeline。
Tooling

工具是微观导演的“镜头语言”

我们不把工具当成花活，而是把它当成“学习可验证”的语言：在 CHECK/CHALLENGE/TRANSFER 等拍点，用最小的工具动作让掌握度变得可测、可追踪。 同时保持宏观导演的克制：对话流动优先，工具只在“必须验证/必须迁移/必须收束”时出现。

CHECK：你说“懂了”，导演要求你证明
出题

主持人用一句话把题“念出来”，你只需要说 A/B/C，系统即时反馈并回主线。

CHALLENGE：挑战者抛反例，逼出边界
反例

不是“更多信息”，而是“更强约束”：帮你把概念从“听懂”推进到“可用”。

TRANSFER：迁移到你真实场景
迁移题

如果你答得流畅，才生成一张“结构图”作为收束；否则返回 MODEL 重建骨架。

关键承诺：工具永远服务于对话推进与学习验收，避免把体验拉回“做题软件”或“内容播放器”。
